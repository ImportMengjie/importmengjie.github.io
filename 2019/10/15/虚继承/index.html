虚继承

Oct 15, 2019

虚继承(virtual inheritance)是多重继承的一种形式, 基类被继承了多次, 但是子类共享该基类的唯一一份副本. 举个例子, Donkey和Horse都继承自Animal基类, 由于众所周知的原因Mule(骡子)通过多继承继承了上边的两个类. 如果是默认情况, 则Mule中含有两个Animal子对象, 分别来自Donkey和Horse. 但是按照逻辑来说, Animal基类中含有的是动物共同的一些特征(如体重,年龄), 甭管是骡子还是马都应该只有一份. 这里就需要虚继承使得Mule只含有一个Animal.

本来写到这里应该结束了, 但是有小朋友问虚继承是如何做到共享基类的呢? 更具体的问题就是虚继承的内存布局是啥样的? 本着送佛送到西的精神, 我想从最简单的继承写起. 开始之前需要说明一点:

由于C++标准并未统一ABI(Application Binary Interface), 如基类, 虚基类在继承类中的位置等内存分布, 所以导致各个编译器的具体实现不同, 这里只写出某种实现,可以实现所需效果.

ABI&API: 它们都是所谓应用程序接口, 只是它们所描述的借口所在层面不一样.

  • API: 指源代码级别的接口, 比如可以说POSIX是一个API标准.
  • ABI: 指二进制层面的接口, 比如C++的对象内存分布是C++ ABI的一部分.

比如POSIX规定了printf()函数原型的API, 但是并未规定参数如何压入栈中等实际运行中二进制级别(ABI)的问题.

以前学到这里的时候爷就有两个疑问, 老师也没说, 爷也不敢问:

  1. 为何非得用指针或是引用才行?

    因为非指针和引用的变量类型是一定的, 绝不可能发生一个变量它类型是基类, 但其实它是派生类对象. 一旦你想将其实现, 你只会得到两个不同的对象, 一个是基类对象, 一个是派生类对象. 因为会调用拷贝构造函数, 重新构建一个新对象给宁.

    指针和引用则可以做到, 基类指针(引用)指向子类对象. 但是引用一生只能引用一次, 指针则可以修改值指向其他对象, 所以指针可能可以更好的发挥多态的优点!

  2. 为何要运行期才能知道调用的哪个函数? 代码是爷写的, 爷肯定知道要调用哪个函数, 那么编译器只要足够聪明不就可以通过代码的上下文推断出具体调用哪个函数嘛, 何必搞这些花里胡哨的玩意.

    虽然代码是自己写的, 但有的时候还真不知道会调用哪个函数, 下面就举个极端的例子:

    如果编译器在编译期就可以确认调用的哪个函数, 那么直接在目标文件中输出 call xxx_function 即可, 而对于运行期才能确认的则不能, 需要特殊的处理, 那么下边介绍如何实现这一特性.

如果要实现这个特性, 那么实例化的对象一定要含有一些信息来确定对其调用虚函数时具体调用哪个函数(是基类的?还是哪个派生类的?), 并且这个信息不应该随着指针类型的变化而”丢失”, 比如说当基类指针指向子类对象时, 如果这个信息在子类的话, 基类指针就不应该能访问到这个信息, 毕竟超出了它的范围. 所以这个信息应该存在基类子对象中.

当然你也不希望多出来的信息占用对象的太多地方, 并且希望这个多出来的信息大小是固定的, 毕竟这是一个多数东西都在编译期确定的语言, 肯定不希望每多个虚函数就要在对象里多加点东西.

那么下边就可以介绍多数编译期采用的虚表和虚表指针实现该特性.

每个class有一个Vtable(Virtual Table), 内含该class之中有作用的virtual function地址, 然后每个object有一个vptr(Virtual table PoinTeR), 指向Vtable的所在. 可以预见编译器会在初始化对象时加入一些操作根据不同类型将vptr指向正确的虚表(Vtable). 而当用指针或者引用调用虚函数时编译器将改造你的代码如下:

这里虚表也含有类的元数据: type_info, 可以实现RTTI(RunTime Type Identification), 在运行时判断类型, 从而实现安全的向下转换等操作. 这告诉我们一个事实, 非多态类不拥有动态判断类型的功能也不应该需要该功能, 因为其不含有任何元数据去描述类型. 而对于拥有多态特性的类来说, 很需要一个安全的向下转换的操作(由基类转换成派生类对象).

我一直有个想法就是不用虚函数去实现多态, 如果没有虚函数就没有元数据, 那么我们可以人为的加一个成员变量来表示它的类型, 再在基类中需要多态的函数里加上判断类型, 调用相应的函数.

上边这样也能实现多态的效果, 但是非常麻烦并且性能也不如直接用虚函数.
